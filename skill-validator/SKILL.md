# 技能验证器

基于 AI 语义分析的技能工作流质量保证工具。

## 目的

此技能使用 AI 代理进行**语义层面的深度验证**,而非简单的关键词匹配,针对以下质量标准验证其他技能:

1. **工作流完整性** - 验证所有步骤已定义且顺序正确
2. **节点转换** - 检查工作流步骤之间的流畅性
3. **人机交互** - 识别哪些步骤需要人工或代理操作
4. **最佳实践** - 根据技能编写指南进行验证

## 验证工作流

### 步骤 1: 读取目标技能

当用户请求验证技能时,读取目标技能的 SKILL.md 文件:

```bash
# 用户会提供技能路径,如:
# "请验证 ~/.claude/skills/my-skill 这个技能"
```

使用 Read 工具读取目标技能的 SKILL.md 内容。

### 步骤 2: 加载验证参考文档

读取本技能的参考文档以了解验证标准:

- `references/validation_checklist.md` - 完整的验证检查清单
- `references/fix_guidance.md` - WHY/WHAT/HOW 修复指导

这些文档定义了所有验证规则和最佳实践。

### 步骤 3: 执行语义验证

根据参考文档,对目标技能进行**语义层面**的深度分析。重点关注:

#### 3.1 结构验证
- 检查 YAML frontmatter 是否包含必需字段(`name`, `description`)
- 验证所有文件引用是否存在
- 检查 description 是否使用第三人称且包含触发短语

#### 3.2 完整性验证
- 识别工作流步骤(不依赖特定关键词格式)
- 检查步骤编号的连续性
- 验证是否有明确的起点和终点
- 评估工作流的完整性和逻辑性

#### 3.3 转换验证
- 分析步骤间的转换是否流畅
- 检查条件分支是否包含所有路径(if/else, 成功/失败)
- 识别可能的死胡同步骤
- 评估转换词的使用(然后、接下来、之后等)

#### 3.4 交互验证
- 区分需要人工干预的步骤和代理自主步骤
- 检查人工输入后是否有验证步骤
- 识别连续的人工步骤(可能需要合并)
- 评估交互模式的清晰度(避免模糊表述)

#### 3.5 最佳实践验证
- 检查 SKILL.md 长度(推荐 <500 行)
- 验证路径格式(使用正斜杠而非反斜杠)
- 检测时效性信息(日期、版本号等)
- 识别过多的替代方案
- 验证关键操作是否有反馈循环

#### 3.6 使用适当性验证
- 检测角色扮演包装反模式(只有人设描述,无工作流)
- 检测纯知识堆砌反模式(只有知识列表,无可执行流程)
- 检查是否存在工具调用(Bash/API/文件操作)
- 评估复用价值(步骤数、任务复杂度)

#### 3.7 认知协议验证
- 检测元问题定义(是否定义深层问题框架而非简单 Q&A 列表)
- 检测追溯推理路径(是否建立"表面问题→诊断→根因→方案"链条)
- 检测输出规范约束(是否约束输出格式强制展示推理过程)
- 检测语义识别深度(是否有多维度问题分类而非单一维度)

### 步骤 4: 生成结构化报告

生成符合以下 JSON Schema 的验证报告:

```json
{
  "skill_path": "技能路径",
  "skill_name": "技能名称",
  "total_issues": 0,
  "critical_count": 0,
  "warning_count": 0,
  "info_count": 0,
  "passed": true,
  "workflow_analysis": {
    "total_nodes": 5,
    "nodes": [
      {
        "step": 1,
        "name": "步骤名称",
        "requires_human": false,
        "has_validation": true
      }
    ],
    "has_start": true,
    "has_end": true
  },
  "interaction_map": {
    "human_required": [
      {"step": 2, "name": "用户提供输入"}
    ],
    "agent_autonomous": [
      {"step": 1, "name": "初始化环境"}
    ],
    "total_steps": 5,
    "human_ratio": 0.2
  },
  "issues": [
    {
      "category": "completeness|transition|interaction|best_practice|structure",
      "severity": "critical|warning|info",
      "title": "问题标题",
      "description": "问题描述",
      "location": "SKILL.md:行号",
      "why": "为什么这是问题?会产生什么影响?",
      "what": "需要做什么来修复?",
      "how": "具体应该如何实施修复?",
      "auto_fixable": false,
      "fix_suggestion": ""
    }
  ]
}
```

#### 严重程度定义

- **critical**: 必须修复才能使技能正常工作
  - 示例:缺失 SKILL.md、缺失必需的 frontmatter 字段、引用的文件不存在
  
- **warning**: 应该修复以提高质量
  - 示例:描述过于简短、人工输入后缺少验证、不完整的条件分支
  
- **info**: 改进建议
  - 示例:缺少转换标记、连续的人工步骤、过多的替代方案

### 步骤 5: 展示验证结果

以清晰易读的格式向用户展示验证结果:

1. **摘要部分**:
   ```
   技能验证报告: my-skill
   ========================================
   状态: [OK] 通过 / [FAIL] 失败
   总问题数: 5
     - 严重: 0
     - 警告: 3
     - 信息: 2
   ```

2. **工作流分析**:
   ```
   工作流分析
   ----------------------------------------
   总步骤数: 5
     步骤 1: 初始化环境 [代理] [V]
     步骤 2: 用户提供输入 [人工] [ ]
     步骤 3: 验证输入 [代理] [V]
     步骤 4: 处理数据 [代理] [ ]
     步骤 5: 生成输出 [代理] [V]
   ```

3. **人机交互映射**:
   ```
   人机交互映射
   ----------------------------------------
   需要人工: 1 步骤
   代理自主: 4 步骤
   人工比率: 20%
   ```

4. **详细问题列表**:
   ```
   [?] [INTERACTION] 输入后无验证
      位置: SKILL.md:45
      用户输入步骤缺少验证
      
      为什么: 未验证的输入会导致下游错误
      是什么: 在人工输入后添加验证步骤
      怎么做: 在步骤 2 后添加验证步骤,检查输入的有效性
   ```

### 步骤 6: 提供修复建议

针对每个问题,根据 `references/fix_guidance.md` 提供具体的修复建议:

- 引用相关的修复模式
- 提供代码/文本示例
- 说明修复的优先级

如果用户请求,可以直接帮助修复简单的问题(如更正 frontmatter、调整路径格式等)。

### 步骤 7: 保存报告(可选)

如果用户请求保存报告,将 JSON 格式的验证结果写入文件:

```json
# 保存到 skill-directory/validation_report.json
```

## 验证类别说明

### 完整性检查

| 检查项 | 严重程度 | 说明 |
|-------|----------|------|
| 缺失 SKILL.md | 严重 | 未找到入口点 |
| 缺失必需字段 | 严重 | frontmatter 缺少 name 或 description |
| 缺失步骤 | 警告 | 步骤序列中存在间隙 |
| 无工作流 | 警告 | 未检测到结构化步骤 |
| 没有明确起点 | 信息 | 缺少明确的起始步骤 |

### 转换检查

| 检查项 | 严重程度 | 说明 |
|-------|----------|------|
| 缺失转换 | 信息 | 步骤之间没有转换标记 |
| 不完整的条件 | 警告 | 有"如果"但没有"否则" |
| 死胡同 | 警告 | 步骤没有后续操作 |

### 交互检查

| 检查项 | 严重程度 | 说明 |
|-------|----------|------|
| 交互模糊 | 警告 | 人工/代理职责不明确 |
| 输入后无验证 | 警告 | 人工输入缺少验证 |
| 连续的人工步骤 | 信息 | 多个人工步骤连续出现 |

### 最佳实践检查

| 检查项 | 严重程度 | 说明 |
|-------|----------|------|
| 简短描述 | 警告 | 描述缺乏具体性 |
| 错误人称 | 警告 | 描述使用第一/第二人称 |
| Windows 路径 | 警告 | 检测到反斜杠路径 |
| 时效性信息 | 警告 | 发现日期特定内容 |
| 选项过多 | 信息 | 提供过多替代方案 |
| SKILL.md 过长 | 警告 | 超过 500 行 |
| 缺失反馈循环 | 警告 | 关键操作缺少验证 |

### 使用适当性检查

| 检查项 | 严重程度 | 说明 |
|-------|----------|------|
| 角色扮演包装 | 警告 | Skill 只有人设描述,无工作流步骤 |
| 纯知识堆砌 | 警告 | Skill 只有知识列表,无可执行流程 |
| 缺少工具调用 | 信息 | Skill 无 Bash/API/文件操作 |
| 复用价值不足 | 信息 | 步骤数 <3 且无工具调用 |

### 认知协议检查

| 检查项 | 严重程度 | 说明 |
|-------|----------|------|
| 元问题定义 | 警告 | 是否定义深层问题框架而非简单 Q&A |
| 追溯推理 | 警告 | 是否建立从问题到根因的推理链 |
| 输出规范 | 信息 | 是否约束输出格式强制展示推理 |
| 语义识别 | 信息 | 是否有多维度问题分类 |

## 使用示例

### 示例 1: 验证新创建的技能

用户: "请帮我验证 ~/.claude/skills/pdf-processor 这个技能"

Agent 执行步骤 1-6,生成并展示验证报告。

### 示例 2: 验证并修复问题

用户: "验证 my-skill 并帮我修复所有 critical 问题"

Agent 执行验证,然后针对严重问题提供修复建议或直接修复。

### 示例 3: 批量验证

用户: "验证 skills 目录下的所有技能"

Agent 遍历目录,对每个技能执行验证,生成汇总报告。

## 重要提示

1. **语义理解优先**: 不要依赖特定的关键词或格式,而是理解内容的语义
2. **参考文档为准**: 所有验证标准以 references 目录下的文档为准
3. **提供具体建议**: 每个问题都必须包含 WHY/WHAT/HOW 三部分
4. **保持报告格式**: 严格遵循 JSON Schema,确保与现有工具兼容
5. **区分严重程度**: 正确分类问题的严重性(critical/warning/info)

## 与其他工具的集成

此技能可以与以下工具配合使用:

- **skill-creator**: 验证后使用 skill-creator 打包技能
- **代码审查工具**: 在技能开发流程中集成验证步骤
- **CI/CD 流程**: 在提交前自动验证技能质量
