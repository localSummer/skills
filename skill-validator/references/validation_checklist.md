# 技能验证检查清单

本文档提供了用于验证技能工作流质量的综合检查清单。基于**语义理解**进行验证,而非简单的关键词匹配。

## 工作流完整性

### 必需元素

- [ ] **明确的起点**: 第一步已清晰定义,明确标记为工作流的开始
- [ ] **顺序步骤**: 所有步骤按逻辑顺序组织,编号连贯
- [ ] **无缺失步骤**: 步骤序列完整,没有逻辑跳跃或间隙
- [ ] **明确的终点**: 最后一步明确标记工作流完成或输出交付
- [ ] **范围定义**: 工作流的输入、输出和边界清晰明确

### 步骤内容要求

- [ ] 每个步骤都有清晰的标题,说明该步骤的主要目的
- [ ] 每个步骤都有具体可执行的操作说明
- [ ] 每个步骤的执行条件和前置要求明确
- [ ] 每个步骤都定义了预期的输出或结果

### 验证方法

通过语义分析识别:
- 步骤的组织结构和编号模式
- 步骤间的逻辑关系和依赖
- 工作流的起点、过程和终点
- 步骤内容的完整性和可执行性

## 节点转换质量

### 转换特征

优秀的步骤转换应该具备:

| 转换类型 | 语义特征 | 质量标准 |
|---------|---------|---------|
| **顺序转换** | 明确说明下一步操作 | 清晰指示执行顺序和流程 |
| **条件转换** | 根据条件选择不同路径 | 所有分支都有明确的处理路径 |
| **前置转换** | 说明执行前的必要条件 | 条件和检查点清晰定义 |
| **完成转换** | 标记步骤完成后的行为 | 明确后续动作或终止条件 |

### 质量标准

- [ ] 每个步骤(除最后一步)都明确说明如何进入下一步
- [ ] 所有条件分支都定义了完整的处理路径(成功和失败)
- [ ] 前置条件在步骤开始前清楚说明
- [ ] 循环或重试逻辑有明确的退出条件

### 反模式

1. **死胡同**: 步骤结束后没有说明下一步操作,工作流中断
2. **缺失替代方案**: 只描述了正常情况,未处理异常或失败情况
3. **隐式转换**: 仅依赖步骤编号暗示顺序,缺乏显式说明
4. **循环引用**: 循环逻辑没有明确的退出条件或终止机制

## 人机交互

### 交互类别

通过语义分析识别步骤的执行主体:

| 类别 | 语义特征 | 判断标准 |
|-----|---------|---------|
| **需要人工输入** | 步骤明确要求用户提供信息或做决策 | 使用明确的人工要求表述 |
| **可选人工输入** | 步骤说明用户可以选择性参与 | 区分必须和可选的人工干预 |
| **代理自主** | 步骤由 AI 代理自动执行 | 没有人工参与要求的处理步骤 |
| **人工验证** | 需要用户确认或审查代理的输出 | 明确的审查、确认或批准要求 |

### 质量标准

- [ ] 每个步骤都清楚地表明由谁执行(人工或代理)
- [ ] 避免使用模糊表述,如"可能需要"、"也许会"
- [ ] 人工输入步骤后有相应的验证或确认步骤
- [ ] 连续的人工步骤有合理的业务理由,或应考虑合并

### 交互模式示例

良好的人机交互模式:
```
步骤 1: [代理] 自动初始化工作环境
步骤 2: [人工] 用户提供配置参数
步骤 3: [代理] 验证用户输入的有效性
步骤 4: [代理] 基于配置执行处理
步骤 5: [人工] 用户审查处理结果
步骤 6: [代理] 根据审查结果最终化
```

## 最佳实践合规性

### Frontmatter 要求

- [ ] 包含 `name` 字段: 小写字母、连字符分隔、简洁明确
- [ ] 包含 `description` 字段: 具体描述功能和使用场景
- [ ] 描述使用第三人称: 避免"我"、"你"等人称代词
- [ ] 描述包含触发条件: 说明何时应该使用此技能

### 内容质量

- [ ] SKILL.md 长度适中(建议少于 500 行)
- [ ] 使用跨平台路径格式(正斜杠)
- [ ] 避免时效性内容(特定日期、版本号等会过期的信息)
- [ ] 术语使用一致,保持统一的命名规范
- [ ] 提供明确建议,避免罗列过多替代方案

### 操作可靠性

- [ ] 关键操作前有明确的验证步骤
- [ ] 错误处理路径完整,说明如何应对失败情况
- [ ] 重要常量有文档说明其含义和选择理由
- [ ] 破坏性操作有确认或回滚机制

### 文档结构

- [ ] 所有引用的文件确实存在
- [ ] 文件引用使用相对路径,深度控制在合理范围
- [ ] 长文档有目录或章节导航
- [ ] 文件名具有描述性,易于理解其用途

## 常见问题及修复

### 问题: 描述缺失使用场景

**问题**: 描述仅说明技能功能,未说明何时使用。

**影响**: 用户和 AI 难以判断何时应该调用此技能。

**修复前**:
```yaml
description: 处理 PDF 文件
```

**修复后**:
```yaml
description: 从 PDF 文件中提取文本和表格,填写表单,合并文档。在处理 PDF 文件、表单填写或文档提取任务时使用。
```

### 问题: 人工交互表述模糊

**问题**: 步骤未明确说明是否需要人工参与。

**影响**: 执行时可能等待用户输入或自动跳过,导致行为不一致。

**修复前**:
```markdown
步骤 3: 可能需要用户确认
```

**修复后**:
```markdown
步骤 3: [人工必需] 等待用户确认后再继续
或
步骤 3: [代理自主] 自动验证并继续,无需用户确认
```

### 问题: 条件分支不完整

**问题**: 只描述了成功路径,未处理失败情况。

**影响**: 遇到错误时工作流中断,无法优雅处理。

**修复前**:
```markdown
步骤 4: 如果验证通过,继续步骤 5
```

**修复后**:
```markdown
步骤 4: 验证用户输入
  - 如果验证通过: 继续步骤 5
  - 如果验证失败: 返回步骤 2,向用户说明错误并请求重新输入
```

### 问题: 关键操作缺少验证

**问题**: 破坏性或不可逆操作前没有确认步骤。

**影响**: 可能造成数据丢失或意外后果。

**修复前**:
```markdown
步骤 5: 删除所有临时文件
```

**修复后**:
```markdown
步骤 5: 列出将要删除的临时文件
步骤 6: [人工] 用户确认删除列表
步骤 7: 执行删除操作
步骤 8: 验证删除结果,确保没有误删重要文件
```

### 问题: 提供过多选项

**问题**: 罗列大量工具或方法,没有明确推荐。

**影响**: 增加决策负担,可能导致选择困难或不一致。

**修复前**:
```markdown
可以使用 pypdf、pdfplumber、PyMuPDF、pdf2image、pdfrw、pikepdf...
```

**修复后**:
```markdown
使用 pdfplumber 提取文本(推荐,兼容性好)。
特殊情况:
- 扫描 PDF 需要 OCR: 使用 pdf2image + pytesseract
- 需要编辑 PDF 结构: 使用 PyMuPDF
```

## 使用适当性检查

**核心原则**: 能用 Prompt 或 Context 搞定的文本,不要强行包装成 Skills。

Skills 的核心价值在于:
- **固化流程**: 把多步骤的复杂操作变成一键触发
- **调用工具**: 整合 Bash 命令、API 调用、文件操作
- **管理上下文**: 处理超长文档、多文件引用

如果 Skill 只是一堆文字描述,没有流程、没有工具调用、没有上下文管理,那它本质上就是一个 Prompt。

### 反模式检测

| 反模式 | 检测信号 | 严重程度 |
|-------|---------|---------|
| 角色扮演包装 | 只有人设/角色描述,无工作流步骤 | warning |
| 纯知识堆砌 | 只有知识列表/规范文档,无可执行操作 | warning |
| 缺少工具调用 | 无 Bash 命令、API 调用、文件操作 | info |
| 复用价值不足 | 步骤数 <3 且无工具调用 | info |

### 判断框架

在判断内容是否适合作为 Skill 时,使用以下三问:

| 问题 | 是 → 考虑 Skill | 否 → 用 Prompt/Context |
|------|----------------|----------------------|
| 是否涉及多步骤流程? | Skill 可以固化流程 | Prompt 更简单 |
| 是否需要调用工具/代码? | Skill 提供确定性执行 | Prompt 足够 |
| 是否会重复使用 3 次以上? | Skill 投入有回报 | 一次性 Prompt |

### 检测方法

**角色扮演包装检测**:
- 检查是否只包含"你是..."、"扮演..."、"作为..."等人设描述
- 验证是否存在编号步骤或工作流定义
- 若只有描述性文本而无执行步骤,标记为反模式

**纯知识堆砌检测**:
- 检查内容是否主要由列表、规范、原则组成
- 验证是否存在可执行的操作指令
- 若内容是"AI 应该知道的知识"而非"AI 应该执行的流程",标记为反模式

**工具调用检测**:
- 搜索 Bash 命令调用(如 `git`、`npm`、`curl` 等)
- 搜索文件操作指令(读取、写入、编辑)
- 搜索 API 调用或外部服务交互
- 若无任何工具调用,标记为潜在问题

**复用价值检测**:
- 统计工作流步骤数量
- 评估任务复杂度
- 若步骤数 <3 且无工具调用,建议使用 Prompt 替代

### 正例与反例

**反例: 角色扮演包装**
```markdown
---
name: senior-architect
description: 以资深架构师的视角回答问题
---
你是一位有 15 年经验的系统架构师,擅长分布式系统设计...
```
问题: 只有人设,无流程。应直接使用 Prompt。

**反例: 纯知识堆砌**
```markdown
---
name: react-best-practices
---
## 组件设计原则
1. 单一职责...
2. 组合优于继承...
## 状态管理最佳实践
...
```
问题: 只有知识,无流程。应放入 CLAUDE.md 或 Context。

**正例: 有效 Skill**
```markdown
---
name: git-commit-push
---
## 工作流
1. 读取暂存区变更
2. 生成 commit message
3. 执行 git commit
4. 执行 git push
```
特点: 有明确步骤、有工具调用、可复用。

## 认知协议验证

**核心理念**: Skill 不是知识库,而是认知协议。它改变的是 AI 如何思考,而非 AI 知道什么。

认知协议包含三层核心要素:

```
Layer 3: 输出规范层 - 约束输出结构和推理展示
         ↑
Layer 2: 追溯推理层 - 建立从表面到根因的推理链
         ↑  
Layer 1: 语义识别层 - 多维识别问题而非关键词匹配
```

### Layer 1: 语义识别（元问题定义）

**核心要求**: 定义"元问题"而非罗列答案

良好表现:
- [ ] 定义了深层的问题框架("问自己"、"为什么"、"追问")
- [ ] 区分不同类型的问题(如学习 vs 工程 vs 架构)
- [ ] 有多维度的分类(如场景 × 问题类型矩阵)
- [ ] 避免纯 Q&A 式的问题列表

反模式:
- 只有"Q: X, A: Y"式的问题-答案对
- 线性的关键词匹配("包含'重渲染'就推荐 memo")
- 无区分地提供相同类型的答案

**检测信号**:
- 搜索反思性表述: "问自己"、"为什么"、"追问"、"核心问题是"
- 检查是否有问题分类框架或决策树
- 验证是否对不同类型问题有差异化处理

### Layer 2: 追溯推理（推理链路）

**核心要求**: 建立"问题 → 诊断 → 原因 → 方案"的完整链

良好表现:
- [ ] 提供"表面问题 vs 根本原因"的映射
- [ ] 解释为什么表面症状背后是特定的原因
- [ ] 从根本原因推导出设计方案(而非快速补丁)
- [ ] 用因果关系连接步骤

反模式:
- 直接给出解决方案,无诊断过程
- 症状 → 快速修复(如"重渲染就加 memo")
- 单向推导,无"为什么"的解释

**检测信号**:
- 搜索因果表述: "因为...所以"、"这表明"、"根本原因是"
- 检查是否有"表面问题 → 根因"的映射表
- 验证方案是否从根因推导而非直接给出

### Layer 3: 输出规范（推理展示）

**核心要求**: 约束输出格式以强制展示推理过程

良好表现:
- [ ] 定义了强制的输出格式(如 WHY/WHAT/HOW)
- [ ] 要求展示推理链或设计决策过程
- [ ] 提供了输出示例
- [ ] 约束了内容结构(如"必须包含"、"顺序是")

反模式:
- 无输出格式约束,回答结构随意
- 只关心答案正确,不关注推理过程
- 无示例说明预期输出

**检测信号**:
- 搜索格式约束: "输出格式"、"必须包含"、"结构化输出"
- 检查是否有输出模板或示例
- 验证是否要求展示推理过程

### 认知协议检验问卷

使用以下问卷快速评估 Skill 是否遵循认知协议:

| 问题 | Layer | 是 | 否 |
|-----|-------|----|----|
| 是否定义了"元问题"或思考框架? | 1 | ☐ | ☐ |
| 是否有多维度问题分类? | 1 | ☐ | ☐ |
| 是否说明"为什么"这是问题? | 2 | ☐ | ☐ |
| 是否提供了表面→根因的推理链? | 2 | ☐ | ☐ |
| 是否约束了输出格式? | 3 | ☐ | ☐ |
| 是否提供了输出示例? | 3 | ☐ | ☐ |

**评分标准**:
- 全部"是": 优秀 - 完整的认知协议
- 5-6 个"是": 良好 - 基本符合认知协议
- 3-4 个"是": 可接受 - 有改进空间
- <3 个"是": 需改进 - 可能是知识库而非认知协议

### 正例与反例

**反例: 知识库型 Skill**
```markdown
---
name: react-performance
---
## 常见问题

Q: 组件为什么重渲染?
A: 可能是状态改变、Props 改变或 Context 改变。

Q: 如何避免不必要的重渲染?
A: 使用 React.memo、useMemo、useCallback。
```
问题: 纯 Q&A 模式,无元问题、无推理链、无输出约束。

**正例: 认知协议型 Skill**
```markdown
---
name: react-performance
---
## 核心问题框架

使用此 Skill 时,先思考:
1. **这是什么类型的问题?**
   - 机制问题(不理解为什么重渲染)?
   - 工程问题(重渲染导致性能问题)?
   - 架构问题(状态设计有缺陷)?

2. **表面症状背后的根因是什么?**

## 问题 → 根因 → 方案 映射

| 表面症状 | 诊断问题 | 根本原因 | 推荐方案 |
|---------|---------|--------|--------|
| 列表项全部重渲染 | 哪个 state 变化触发? | 整个列表 state 在父组件 | 状态下沉 |

## 输出规范 (强制要求)

每个回答必须包含:
### 推理链
+-- Layer 1: [表面问题]
|       ↑
+-- Layer 3: [设计原则]
|       ↓
+-- Layer 2: [根本原因]

### 推荐方案
[基于推理的方案]
```
特点: 有元问题框架、有推理链、有输出约束。

## 验证原则

1. **语义理解优先**: 通过理解内容的含义和意图进行验证,而非匹配特定词语
2. **上下文分析**: 考虑步骤在整个工作流中的位置和作用
3. **逻辑连贯性**: 验证步骤间的逻辑关系是否合理、完整
4. **用户视角**: 从技能使用者的角度评估清晰度和可用性
5. **鲁棒性检查**: 确保异常情况和边界条件都有合理处理
6. **使用适当性**: 验证内容是否真正需要 Skill 形式,而非 Prompt 或 Context
