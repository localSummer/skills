# 修复指南 - Why/What/How 框架

本文档使用 Why/What/How 思维框架,为修复技能验证问题提供结构化指导。基于**语义理解**而非关键词匹配。

## 框架概述

每个问题修复都遵循三个问题:

1. **WHY(为什么)** - 为什么这是问题?会产生什么影响?
2. **WHAT(是什么)** - 需要做什么来修复它?
3. **HOW(怎么做)** - 究竟应该如何实施修复?

该框架确保:
- 在行动前理解根本原因
- 明确的目标定义
- 具体的实施步骤

## 结构问题

### 缺失 SKILL.md

**WHY**: SKILL.md 是必需的入口点。没有它,技能无法加载或被发现。

**WHAT**: 创建包含适当 YAML frontmatter 和 markdown 指令的 SKILL.md 文件。

**HOW**:

创建基本的 SKILL.md 结构:

```markdown
---
name: your-skill-name
description: 描述此技能的作用以及何时使用它。
---

# 技能标题

## 目的

解释此技能完成什么任务,为什么需要它。

## 工作流

### 步骤 1: 第一个操作

详细说明如何执行第一步...

### 步骤 2: 第二个操作

详细说明如何执行第二步...
```

关键要素:
- YAML frontmatter 必须在文件开头
- name 和 description 是必需字段
- 使用清晰的章节结构组织内容

### 缺失 Frontmatter 字段

**WHY**: `name` 字段标识技能;`description` 启用发现功能。缺少这些字段会导致技能无法被正确识别和调用。

**WHAT**: 将缺失的字段添加到 YAML frontmatter。

**HOW**:

确保 frontmatter 包含所有必需字段:

```yaml
---
name: your-skill-name    # 小写字母、连字符分隔、简洁明确
description: >
  处理 X 并生成 Y。当用户询问 X、提到 Y 或需要 Z 功能时使用。
  包含具体的使用场景和触发条件。
---
```

描述最佳实践:
- 第一句说明技能做什么
- 第二句说明何时使用它
- 使用第三人称
- 包含关键触发词

### 缺失引用的文件

**WHY**: 死链接会导致执行失败并让 Claude 感到困惑。用户点击链接时会遇到 404 错误。

**WHAT**: 创建缺失的文件或删除引用。

**HOW**:

检查方法:
1. 阅读 SKILL.md 找出所有文件引用
2. 检查每个引用的文件是否存在
3. 对于缺失的文件,决定是创建它还是删除引用

常见引用模式:
- Markdown 链接: `[text](path/to/file.md)`
- 脚本引用: `python scripts/example.py`
- 相对路径: `references/guide.md`

修复选项:
- **创建文件**: 如果引用的内容确实需要,创建该文件并填充相应内容
- **删除引用**: 如果引用的内容不必要,删除该引用
- **修正路径**: 如果文件存在但路径错误,更正路径

## 完整性问题

### 无结构化工作流

**WHY**: 没有清晰的步骤,Claude 可能会遗漏程序或错误执行,导致不一致的结果。

**WHAT**: 添加带有清晰结构的编号工作流步骤。

**HOW**:

组织工作流的最佳方式:

```markdown
## 工作流

### 步骤 1: 初始化

准备必要的环境和资源:
- 检查依赖项
- 加载配置
- 初始化状态

### 步骤 2: 处理

执行主要逻辑:
- 读取输入
- 转换数据
- 应用业务规则

### 步骤 3: 验证

确保结果正确:
- 检查输出格式
- 验证数据完整性
- 确认无错误

### 步骤 4: 完成

收尾工作:
- 保存结果
- 清理临时资源
- 返回输出
```

关键原则:
- 每个步骤有明确的目的
- 步骤按逻辑顺序排列
- 每个步骤的输入输出清晰
- 包含验证和错误处理

### 序列中缺失步骤

**WHY**: 编号中的间隙会让 Claude 对工作流顺序感到困惑,可能跳过重要操作。

**WHAT**: 添加缺失的步骤或重新编号现有步骤。

**HOW**:

诊断:检查步骤编号是否连续(1, 2, 3, 4...)

修复方案一 - 重新编号:
```markdown
# 修复前(步骤 3 缺失)
步骤 1: 初始化
步骤 2: 处理
步骤 4: 完成

# 修复后(重新编号)
步骤 1: 初始化
步骤 2: 处理
步骤 3: 完成
```

修复方案二 - 补充步骤:
```markdown
# 修复前(缺少验证步骤)
步骤 1: 初始化
步骤 2: 处理
步骤 4: 完成

# 修复后(添加遗漏的验证)
步骤 1: 初始化
步骤 2: 处理
步骤 3: 验证处理结果  # <-- 新增
步骤 4: 完成
```

选择标准:
- 如果步骤确实没有遗漏逻辑,选择重新编号
- 如果发现确实缺少必要步骤,补充该步骤

### 没有明确的开始/结束

**WHY**: 边界不清晰会让 Claude 不确定从哪里开始/结束,可能执行不完整或过度执行。

**WHAT**: 明确标记起点和终点。

**HOW**:

在第一步明确标记开始:

```markdown
### 步骤 1: 开始 - 初始化环境

**这是工作流的起点**

执行以下初始化操作:
1. 检查必需的依赖项
2. 加载配置文件
3. 设置工作目录

**前置条件**: 无(这是第一步)
**完成后**: 继续步骤 2
```

在最后一步明确标记结束:

```markdown
### 步骤 5: 完成 - 交付结果

**这是工作流的终点**

执行最终操作:
1. 整理输出内容
2. 向用户展示结果
3. 清理临时文件

**完成后**: 工作流结束,无需进一步操作
```

## 转换问题

### 缺失转换标记

**WHY**: 没有明确的转换说明,步骤间的衔接不清晰,可能导致执行顺序混乱。

**WHAT**: 在步骤之间添加明确的转换说明。

**HOW**:

在每个步骤末尾添加转换指导:

```markdown
### 步骤 2: 处理数据

执行数据转换和处理...

[处理逻辑]

**完成此步骤后**,继续进行验证(步骤 3)。

---

### 步骤 3: 验证输出

**开始前确认**: 步骤 2 已成功完成且有输出数据。

验证数据的正确性...
```

转换类型:
- **顺序转换**: "完成后,继续步骤 X"
- **条件转换**: "如果 Y,则执行步骤 X;否则执行步骤 Z"
- **前置要求**: "在开始前,确保 X 已完成"

### 不完整的条件分支

**WHY**: 缺少替代路径会让 Claude 在遇到非预期情况时不知所措,可能导致工作流中断。

**WHAT**: 为每个条件定义完整的处理路径(成功和失败)。

**HOW**:

完整的条件分支模式:

```markdown
### 步骤 3: 验证输入

对用户提供的数据进行验证:
- 检查格式是否正确
- 确认必填字段存在
- 验证数据有效性

**验证结果处理**:

如果**验证通过**:
- 记录验证成功
- 继续步骤 4: 处理数据

如果**验证失败**:
- 记录具体的错误信息
- 向用户展示错误详情
- 返回步骤 2: 用户重新输入
- 提供修正建议
```

关键要素:
- 明确列出所有可能的结果
- 每个结果都有明确的后续动作
- 说明如何从错误中恢复

### 死胡同步骤

**WHY**: 步骤结束后没有后续指示,工作流会中断,用户不知道接下来该做什么。

**WHAT**: 为每个步骤(除最后一步)添加明确的后续动作。

**HOW**:

确保每个步骤都有出口:

```markdown
### 步骤 4: 生成报告

根据处理结果生成详细报告:
- 汇总统计信息
- 生成可视化图表
- 编写说明文档

**报告生成完成后**:
- 将报告保存到指定位置
- 继续步骤 5: 交付结果

---

### 步骤 5: 交付结果

将报告交付给用户:
- 显示报告摘要
- 提供下载链接
- 询问是否需要进一步操作

**这是最后一步,完成后工作流结束**
```

## 交互问题

### 人工需求模糊

**WHY**: 职责不明确会导致行为不一致,有时等待用户输入,有时自动继续,造成混乱。

**WHAT**: 明确标记每个步骤是需要人工还是代理执行。

**HOW**:

使用明确的标记:

```markdown
# 修复前(模糊)
步骤 3: 可能需要用户确认
步骤 4: 根据情况决定是否询问用户

# 修复后(明确 - 需要人工)
步骤 3: [需要人工] 等待用户确认配置

用户必须审查以下配置并确认:
- 目标目录: /path/to/output
- 处理模式: 批量处理
- 覆盖现有文件: 是

**用户确认后**,继续步骤 4。

# 修复后(明确 - 代理自主)
步骤 3: [代理自主] 自动验证配置

系统自动检查配置的有效性:
- 验证路径存在且可写
- 确认处理模式有效
- 检查文件冲突

**无需用户干预**,验证通过后自动继续步骤 4。
```

明确性指南:
- 使用 `[需要人工]` 或 `[代理自主]` 标记
- 避免"可能"、"也许"、"根据情况"等模糊词
- 说明等待或自动继续的具体条件

### 人工输入后缺失验证

**WHY**: 未验证的输入会导致后续步骤失败,产生级联错误,浪费时间和资源。

**WHAT**: 在每个人工输入步骤后添加验证步骤。

**HOW**:

人工输入 + 验证的标准模式:

```markdown
### 步骤 2: [人工] 用户提供配置

请用户提供以下信息:
- 输入文件路径
- 输出格式(JSON/CSV/XML)
- 处理选项

---

### 步骤 3: [代理] 验证用户输入

**在继续处理前**,验证用户提供的信息:

验证检查:
1. 文件路径验证
   - 文件是否存在
   - 是否有读取权限
   - 文件格式是否支持

2. 格式验证
   - 输出格式是否在支持列表中
   - 格式与输入类型是否兼容

3. 选项验证
   - 选项组合是否有效
   - 参数值是否在合理范围

**如果验证失败**:
- 向用户显示具体错误
- 返回步骤 2 请求重新输入
- 提供正确的输入示例

**如果验证成功**:
- 继续步骤 4: 执行处理
```

验证要点:
- 在使用输入前立即验证
- 检查所有关键属性
- 提供清晰的错误消息
- 允许用户修正输入

### 连续的人工步骤

**WHY**: 频繁的中断会降低效率和用户体验,可能表明步骤设计不够合理。

**WHAT**: 合并相关的人工输入步骤,或在其间插入代理处理步骤。

**HOW**:

方案一 - 合并相关输入:

```markdown
# 修复前(连续人工步骤)
步骤 2: [人工] 用户提供姓名
步骤 3: [人工] 用户提供邮箱
步骤 4: [人工] 用户提供电话

# 修复后(合并为一个步骤)
步骤 2: [人工] 用户提供联系信息

请用户一次性提供完整的联系信息:
- 姓名(必填)
- 邮箱(必填)
- 电话(可选)
- 公司(可选)

**所有信息收集完成后**,继续步骤 3。
```

方案二 - 插入代理步骤:

```markdown
# 修复前(连续人工)
步骤 2: [人工] 用户选择模板
步骤 3: [人工] 用户填写内容

# 修复后(插入验证)
步骤 2: [人工] 用户选择模板

步骤 3: [代理] 加载并预览模板
- 读取模板文件
- 展示模板结构
- 说明需要填写的字段

步骤 4: [人工] 用户填写模板内容
```

## 最佳实践问题

### 描述未使用第三人称

**WHY**: 第一/第二人称会导致技能发现不一致,描述读起来像对话而非规范说明。

**WHAT**: 用第三人称客观描述技能功能。

**HOW**:

改写技巧:

```yaml
# 修复前(第一/第二人称)
description: 我可以帮你处理 PDF 文件,你可以用我来提取文本

# 修复后(第三人称)
description: 处理 PDF 文件,提取文本和表格。在需要处理 PDF 文档时使用。
```

常见转换模式:
- "我可以" → "可以" 或 "能够"
- "我会" → "将" 或直接描述功能
- "你可以" → "允许" 或 "支持"
- "帮你" → "帮助" 或删除
- "你的文件" → "文件" 或 "用户的文件"

重写原则:
- 客观描述功能,而非对话
- 聚焦于技能做什么,而非谁用它
- 保持专业和规范的语气

### SKILL.md 过长

**WHY**: 长文件会消耗过多的上下文令牌,影响 AI 处理效率,也不利于阅读和维护。

**WHAT**: 将内容拆分为主文件 SKILL.md 和多个参考文件。

**HOW**:

拆分策略:

```markdown
# SKILL.md(保留核心内容)
---
name: my-skill
description: 技能描述
---

# 技能名称

## 目的
简要说明技能用途(2-3 段)

## 快速开始
基本使用方法

## 工作流
核心步骤(简化版)

### 步骤 1: 初始化
简要说明...
详细内容参见 [references/step1_detail.md](references/step1_detail.md)

### 步骤 2: 处理
简要说明...
详细内容参见 [references/step2_detail.md](references/step2_detail.md)

## 参考资料
- [详细工作流](references/detailed_workflow.md)
- [API 参考](references/api_reference.md)
- [常见问题](references/faq.md)
- [示例集合](references/examples.md)
```

拆分原则:
- SKILL.md 保留概述和核心流程
- 详细说明、API 文档、示例移到 references/
- 通过链接连接主文件和参考文件
- 每个参考文件聚焦单一主题

### 缺失反馈循环

**WHY**: 没有验证的关键操作可能会造成不可逆的损害,如数据丢失、错误部署等。

**WHAT**: 为关键操作添加计划-验证-执行-确认的完整循环。

**HOW**:

完整的反馈循环模式:

```markdown
### 步骤 4: 计划变更

[代理] 分析并生成变更计划:
- 扫描需要修改的文件
- 生成变更清单
- 评估影响范围
- 创建 `changes_plan.json`

**重要**: 此步骤只生成计划,不执行任何修改。

---

### 步骤 5: [人工] 审查变更计划

用户必须审查变更计划:
- 查看将要修改的文件列表
- 确认变更内容符合预期
- 评估潜在风险

用户决定:
- **批准**: 继续步骤 6
- **拒绝**: 终止工作流
- **修改**: 返回步骤 4 调整计划

---

### 步骤 6: [代理] 执行变更

应用已批准的变更:
```bash
python apply_changes.py changes_plan.json
```

记录执行日志到 `execution.log`

---

### 步骤 7: [代理] 验证结果

确认变更已正确应用:
1. 检查所有文件是否成功修改
2. 运行自动化测试
3. 验证系统功能正常

**如果验证失败**:
- 自动回滚变更
- 记录错误详情
- 通知用户并说明失败原因

**如果验证成功**:
- 确认变更完成
- 生成成功报告
- 继续步骤 8(如有)
```

关键原则:
- 计划与执行分离
- 关键操作前需要人工确认
- 执行后立即验证结果
- 提供回滚机制
- 完整的日志记录

## 使用适当性问题

### 角色扮演/知识堆砌反模式

**WHY**:
- Skill 的核心价值是"固化流程、调用工具、管理上下文"
- 纯文本描述(角色设定、知识规范)不需要 Skill 的触发机制
- 包装成 Skill 增加维护成本,却无额外价值
- 占用 Skill 目录的认知空间,降低真正有效 Skill 的触发率

**WHAT**:
根据内容类型选择合适的载体:
- 角色设定 → 直接写入 Prompt 或 CLAUDE.md
- 知识规范 → 放入项目 Context 文件或 CLAUDE.md
- 有流程的 → 保留为 Skill,但需补充工作流步骤

**HOW**:

**角色扮演的处理**:

```markdown
# 差: 包装成 Skill
---
name: senior-architect
---
你是一位有 15 年经验的系统架构师...

# 好: 直接用 Prompt
"请以资深架构师的视角,分析这个设计方案"

# 好: 放入 CLAUDE.md
## 默认角色
以资深架构师的视角进行技术讨论...
```

**知识堆砌的处理**:

```markdown
# 差: 包装成 Skill
---
name: react-best-practices
---
## 组件设计原则
1. 单一职责...
...(大量规范文档)

# 好: 放入 CLAUDE.md 或项目 Context
# 在 CLAUDE.md 中
## React 开发规范
遵循以下原则:
1. 单一职责...
```

**保留为 Skill 的改造**:
如果内容确实需要固化为流程,添加工作流步骤:

```markdown
---
name: react-component-review
description: 审查 React 组件代码质量
---
## 工作流
1. 读取目标组件文件
2. 检查单一职责原则
3. 检查状态管理模式
4. 生成审查报告
5. 提供改进建议
```

### 缺少工具调用/复用价值不足

**WHY**:
- Skills 的确定性价值来自工具调用(Bash、文件操作、API)
- 无工具调用的 Skill 与 Prompt 效果相同
- 步骤过少(<3步)的任务投入产出比低
- 不会重复使用 3 次以上的任务不值得固化

**WHAT**:
评估 Skill 的必要性:
1. 若有工具调用需求 → 保留并完善
2. 若无工具调用且步骤 <3 → 考虑转为 Prompt
3. 若确实需要复杂流程 → 添加更多步骤和细节

**HOW**:

**判断流程图**:
```
是否涉及多步骤流程? ─否→ 用 Prompt
        │
       是
        ↓
是否需要调用工具/代码? ─否→ 用 Prompt/Context
        │
       是
        ↓
是否会重复使用 3 次以上? ─否→ 一次性 Prompt
        │
       是
        ↓
     保留为 Skill
```

**添加工具调用**:

```markdown
# 修复前: 无工具调用
## 工作流
1. 分析需求
2. 设计方案
3. 输出结果

# 修复后: 添加工具调用
## 工作流
1. 读取项目结构(使用 ls/tree)
2. 分析现有代码模式(使用 grep/read)
3. 设计方案并写入文件
4. 验证输出(使用 Bash 执行检查)
```

**精简为 Prompt 的情况**:
如果任务确实简单,直接删除 Skill,用 Prompt 替代:
```
# 删除 Skill: simple-greeting
# 替代方案: 直接在对话中说
"请用友好的语气打招呼"
```

## 认知协议问题

### 缺少元问题定义（纯 Q&A 模式）

**WHY**:
- 元问题定义能引导 Claude 进行深度思考,而不是简单的关键词匹配
- 纯 Q&A 模式使 Skill 沦为"知识库",失去认知协议的价值
- 没有元问题,AI 只会做表面的问题-答案匹配,无法举一反三
- 用户也无法从 Skill 中学到思考方法,只能得到具体答案

**WHAT**:
添加"核心问题框架"或"元问题"部分,定义使用此 Skill 时应该思考的深层问题。

**HOW**:

```markdown
# 修复前(纯 Q&A 模式)
## 常见问题

Q: 组件为什么重渲染?
A: 可能是状态改变、Props 改变或 Context 改变。

Q: 如何避免不必要的重渲染?
A: 使用 React.memo、useMemo、useCallback。

# 修复后(添加元问题框架)
## 核心问题框架

使用此 Skill 时,先思考以下问题:

1. **这是什么类型的问题?**
   - 机制问题(不理解为什么重渲染)? → 需要解释原理
   - 工程问题(重渲染导致性能问题)? → 需要诊断和优化
   - 架构问题(状态设计有缺陷)? → 需要重新设计

2. **表面症状背后的根因是什么?**
   - 不要急于给出解决方案
   - 先追问"为什么会出现这个症状"

3. **修复方案的优先级**
   - 先考虑设计层修复(状态下沉/拆分)
   - 再考虑优化层补充(memo/useMemo)
   - 最后才是 workaround(debounce 等)

## 常见问题

(以上面的框架指导回答每个问题)
```

**关键改进**:
- 前置问题框架,引导深度思考
- 问题按"类型"而非"关键词"组织
- 明确修复优先级,避免直接给答案

### 缺少追溯推理路径

**WHY**:
- 没有"表面问题 → 根本原因"的推理链,AI 只能做表面修复
- 真正的专家知识在于识别症状背后的根因,然后从根因推导方案
- 缺少追溯路径,用户无法学会举一反三
- AI 可能给出"语法正确但设计错误"的方案

**WHAT**:
建立"问题 → 诊断 → 原因 → 方案"的完整推理链。

**HOW**:

```markdown
# 修复前(直接给方案)
## 性能优化方案

- 重渲染 → 用 React.memo
- deps 警告 → 加上依赖
- props drilling → 用 Context

# 修复后(建立推理链)
## 性能问题诊断和修复

### 问题 → 根因 → 方案 映射表

| 表面症状 | 诊断问题 | 根本原因 | 推荐方案 |
|---------|---------|--------|--------|
| 列表组件每项都重渲染 | 哪个 state 变化触发了? | 整个列表 state 在父组件,子组件接收全部数据 | 状态下沉:把列表项 state 移到子组件 |
| input 输入卡顿 | 为什么 input 值变化会重渲染整个页面? | 表单 state 与页面其他 state 耦合 | 状态拆分:表单 state 独立管理 |
| useEffect 依赖循环 | 这个副作用的边界在哪里? | 副作用混合了多个关注点 | 职责分离:一个副作用只负责一件事 |

### 修复流程

对于任何性能问题,按以下顺序思考:

1. **识别根因**(先别急着加 memo)
   - 哪个 state 变化导致了问题?
   - 这个 state 为什么放在这里?
   - 有多少组件实际依赖它?

2. **设计层修复**(根据根因)
   - 能否通过状态下沉避免不必要传递?
   - 能否通过状态拆分降低耦合?

3. **优化层补充**(只在必要时)
   - 确认设计无缺陷后,再考虑加 memo
```

**关键改进**:
- 提供"症状 → 根因"的映射表
- 解释为什么某个症状对应某个根因
- 从根因推导方案,而非直接给答案
- 明确修复流程的优先级

### 缺少输出规范约束

**WHY**:
- 无输出格式约束,AI 的回答结构随意,质量不稳定
- 约束输出格式能强制 AI 展示推理过程,这本身就是一种教学
- 用户可以从推理过程中学习,而不仅仅得到答案
- 结构化输出便于复用和验证

**WHAT**:
添加"MANDATORY OUTPUT FORMAT"部分,约束每个回答必须包含推理链。

**HOW**:

```markdown
# 修复前(无输出约束)
(回答结构随意,有时只给代码,有时只给解释)

# 修复后(添加输出规范)
## 输出规范 (强制要求)

每个回答必须遵循以下结构:

### 1. 推理链

展示从问题到方案的完整推理:

```
+-- Layer 1: [用户的表面问题]
|       ↑
+-- Layer 3: [相关的设计原则/约束]
|       ↓
+-- Layer 2: [识别的根本原因]
```

### 2. 设计约束分析

列出相关的设计原则或场景约束,支撑诊断:
- [原则 1]
- [原则 2]

### 3. 推荐方案

基于诊断和约束,提出设计方案:
```code
// 方案代码
```

### 4. 方案说明

解释为什么这个方案能解决根本原因(而非只是消除症状)。

---

**输出示例**:

### 推理链
+-- Layer 1: 列表中每一项都会重渲染
|       ↑
+-- Layer 3: React 推崇单向数据流和最小化 state
|       ↓
+-- Layer 2: 整个列表 state 在父组件,子组件接收全部引用

### 设计约束分析
- React 状态管理原则: state 应存储在最需要它的组件
- 性能优化原则: 减少重渲染的最优方案是减少 state 更新的范围

### 推荐方案
```jsx
// 状态下沉: 把列表项的 state 移到子组件
function ListItem({ id, initialValue }) {
  const [value, setValue] = useState(initialValue);
  // ...
}
```

### 方案说明
通过状态下沉,每个列表项的状态更新只影响自己,
不会触发其他列表项重渲染。这是从设计层面解决问题,
而非简单地加 memo 来掩盖设计缺陷。
```

**关键改进**:
- 定义了强制的输出结构
- 要求展示推理链
- 提供了具体的输出示例
- 约束了内容组织顺序

### 语义识别不足（单一维度）

**WHY**:
- 单一维度的问题分类(如只按关键词)无法区分相似问题的不同本质
- 同一个表面症状可能有完全不同的根因,需要不同的处理方式
- 多维度分类能帮助 AI 理解问题的不同侧面
- 用户也能从分类中学会如何分析问题

**WHAT**:
引入多维度分类矩阵,按问题类型、场景、根因等多个维度组织内容。

**HOW**:

```markdown
# 修复前(单一维度)
## 常见问题

- useEffect 问题 → [一堆内容]
- 性能问题 → [一堆内容]
- 状态管理问题 → [一堆内容]

# 修复后(多维度分类)
## 多维度问题分类

### 维度 1: 问题本质

同样包含"useEffect"关键词的问题,本质可能完全不同:

| 本质 | 特征 | 解决方式 |
|-----|------|--------|
| 机制理解 | "useEffect 怎么用?" | 讲解机制 + 示例 |
| 性能工程 | "useEffect 导致页面卡" | 诊断原因 + 架构改进 |
| 设计缺陷 | "useEffect 依赖循环,加了依赖就死循环" | 重设计数据流 |

### 维度 2: 场景特征

不同场景有不同的约束和最佳实践:

| 场景 | 特殊约束 | 推荐做法 |
|-----|--------|--------|
| 表单输入 | 高频更新、需要及时反馈 | 本地 state + 防抖上报 |
| 列表渲染 | 数据量可能很大、需要虚拟化 | 状态规范化 + 选择性订阅 |
| 实时数据 | 需要 WebSocket 订阅、数据源多 | 分离数据层和 UI 层 |

### 组合应用

当遇到新问题时:
1. 先按"问题本质"判断应该讲解/诊断/重设计
2. 再按"场景特征"应用相应的最佳实践
3. 最后才按具体问题选择工具(memo/useMemo/等)

### 决策树

```
收到问题
    ↓
识别问题本质
    ├── 机制理解 → 解释原理 + 示例
    ├── 性能工程 → 诊断 + 设计层修复
    └── 设计缺陷 → 追溯根因 + 重新设计
            ↓
        识别场景特征
            ├── 表单场景 → 应用表单最佳实践
            ├── 列表场景 → 应用列表最佳实践
            └── 实时数据 → 应用实时数据最佳实践
```
```

**关键改进**:
- 引入多维度分类(问题本质 × 场景特征)
- 提供分类矩阵和决策树
- 解释不同维度如何组合应用
- 帮助 AI 和用户都能更准确地定位问题
