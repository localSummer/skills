# Skill 迭代复盘（大前端开发版）

回顾当前对话，整理 skill 使用情况、代码质量反馈和迭代建议。

---

## 触发条件

- `skill复盘`（主命令）
- `Skill迭代清单`（别名）

---

## 工作流程

### 第一步：回顾对话

扫描当前对话的全部内容，识别：
1. 调用了哪些 skill
2. 每个 skill 产出了什么内容
3. 用户对产出的反馈（正面/负面）

### 第二步：提取负反馈

找出用户表达不满或修正意见的地方：

**通用反馈：**
- 直接否定：「不对」「不行」「这个不好」
- 修正意见：「应该是...」「改成...」「其实是...」
- 质疑：「真的吗？」「这样对吗？」
- 补充要求：「还要加上...」「漏了...」

**前端开发特定反馈：**
- 代码质量：「这里没有类型检查」「缺少错误处理」「性能有问题」
- 架构设计：「组件拆分不合理」「状态管理太复杂」「这个应该抽成 hook」
- 最佳实践：「没有遵循 React 规范」「这里会造成内存泄漏」「应该用 useMemo 优化」
- 用户体验：「加载状态没处理」「错误提示不友好」「响应式布局有问题」
- 工程规范：「命名不符合规范」「文件结构不清晰」「缺少单元测试」

记录：
- 具体是哪个 skill 的产出
- 用户的原话
- 问题的本质是什么

### 第三步：分析迭代方向

对每条负反馈，判断：

**A. 需要更新现有 Skill**
- 新增检查点/规则
- 修改输出格式
- 调整流程步骤
- 补充注意事项

**B. 需要新建 Skill**
- 发现了新的工作流程
- 现有 skill 覆盖不了的场景
- 可复用的新方法论

**C. 不需要改动**
- 一次性的特殊情况
- 用户个人偏好，不具普遍性

### 第四步：生成迭代清单

整理成待确认的行动清单。

## 判断标准

### 什么情况建议「更新现有 Skill」

- 用户反馈指向 skill 的某个具体步骤或规则
- 问题可以通过增加检查点解决
- 问题在该 skill 的职责范围内

**前端开发示例：**
- 「react-developer 生成的组件缺少 PropTypes 或 TypeScript 类型」→ 更新 react-developer，增加类型定义检查点
- 「code-review 没有检查 useEffect 的依赖数组」→ 更新 code-review，增加 React Hooks 规则检查
- 「组件命名不符合项目规范」→ 更新相关 skill，增加命名规范检查

### 什么情况建议「新建 Skill」

- 用户描述了一个完整的新工作流
- 现有 skill 都不覆盖这个场景
- 这个方法论可以复用（不是一次性的）
- 用户明确说「这个应该做成一个 skill」

**前端开发示例：**
- 「需要一个专门分析组件性能瓶颈的工具」→ 新建 performance-analyzer skill
- 「希望有个工具能自动生成 Storybook stories」→ 新建 storybook-generator skill
- 「需要检查组件的可访问性问题」→ 新建 a11y-checker skill
- 「想要一个工具自动重构 Class 组件到 Function 组件」→ 新建 class-to-hooks skill

### 什么情况「不需要改动」

- 用户的反馈是针对具体内容，不是针对方法
- 属于个人偏好，不具普遍性
- 一次性的特殊情况

**前端开发示例：**
- 「这个变量名改成 userData 比较好」→ 具体内容建议，不需要改动 skill
- 「我个人喜欢用箭头函数」→ 个人偏好，不需要改动
- 「这次需要兼容 IE11」→ 一次性特殊需求，不需要改动

## 前端开发常见 Skill 迭代场景

### 代码质量提升

**触发场景：**
- 生成的代码缺少类型检查
- 错误处理不完善
- 内存泄漏隐患

**迭代方向：**
- 更新 react-developer/code-review：增加 TypeScript 类型定义规范
- 更新相关 skill：增加错误边界检查点
- 新建 memory-leak-detector skill：专门检测内存泄漏

### 性能优化

**触发场景：**
- 大列表渲染卡顿
- 频繁的不必要重渲染
- 首屏加载缓慢

**迭代方向：**
- 更新 react-developer：增加虚拟滚动建议
- 更新 code-review：增加 React.memo/useMemo 检查
- 新建 performance-optimizer skill：分析性能瓶颈

### 架构重构

**触发场景：**
- 组件过于庞大
- 状态管理混乱
- 逻辑复用困难

**迭代方向：**
- 更新 react-developer：增加组件拆分指导
- 新建 state-management-refactor skill：重构状态管理
- 新建 hook-extractor skill：提取自定义 hook

### 工程规范

**触发场景：**
- 命名不一致
- 文件结构混乱
- 缺少单元测试

**迭代方向：**
- 更新所有 skill：统一命名规范检查
- 新建 project-structure-optimizer skill：优化文件结构
- 更新 code-review：增加测试覆盖率检查

### 用户体验

**触发场景：**
- 缺少加载状态
- 错误提示不友好
- 交互反馈不及时

**迭代方向：**
- 更新 react-developer：增加 UX 检查清单
- 新建 ux-enhancer skill：优化用户体验
- 更新相关 skill：增加可访问性检查

## 注意事项

### 通用注意事项
1. 只整理当前对话的内容，不跨对话
2. 负反馈要记录用户原话，保留上下文
3. 迭代建议要具体，不能太抽象
4. 新 skill 建议要说明来源（哪段对话触发的）
5. 优先级判断：影响产出质量的 > 影响效率的 > 影响格式的

### 前端开发特定注意事项
1. **代码质量优先**：影响运行时正确性、类型安全、内存泄漏的问题优先级最高
2. **性能问题分级**：
   - 高优先级：严重影响用户体验的性能问题（如大列表卡顿、频繁重渲染）
   - 中优先级：可感知但不严重的性能问题（如首屏加载稍慢）
   - 低优先级：理论上的性能优化（如过度的 memo）
3. **架构设计问题**：组件拆分、状态管理、代码复用等架构问题应优先考虑重构
4. **工程规范**：命名、文件结构、注释等规范问题优先级较低，但应批量处理
5. **用户体验**：加载状态、错误提示、交互反馈等 UX 问题应优先处理
6. **测试覆盖**：缺少单元测试的核心逻辑应标注为中优先级
7. **可访问性**：ARIA 标签、键盘导航等 a11y 问题应根据项目要求判断优先级
